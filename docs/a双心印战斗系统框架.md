# 《双心印》战斗系统框架

本文档用于给主程序明确《双心印》战斗系统的框架内容：当前以单角色本地验证为验收口径，但整体结构必须服务未来双人共斗与联机扩展，且不跳出既定 `GAS` 技术路线。本文档只定义战斗系统的模块划分、结构关系、状态与验收边界，不展开具体角色、具体招式数值、具体特效资源与具体代码实现。

## 1. 战斗系统模块划分

当前战斗系统框架拆分为四个一级模块：

1. 基础战斗模块
2. 技能系统模块（结印系统）
3. 合印系统模块
4. 战斗数据与状态模块

其中：

- 基础战斗模块负责玩家最常使用的动作循环
- 技能系统模块负责角色主动战斗能力的核心差异化
- 合印系统模块负责双心印最核心的双人协同特色
- 战斗数据与状态模块负责支撑上述系统运行的统一状态表达

## 2. 基础战斗模块

### 2.1 模块目标

基础战斗模块用于支撑玩家最基本的动作体验，是所有技能与协同玩法的底座。

### 2.2 模块组成

基础战斗模块当前包含以下内容：

1. 受攻击与受击反馈
2. 普通攻击
3. 预输入
4. 闪避
5. 格挡

在阅读顺序上，本章优先定义受击这一类底层反馈规则，再说明玩家最基础的输出方式，随后说明输入缓存机制，最后说明两种主动防御动作。这样可以让后续各系统之间的引用关系更清晰。

另外，本章已经频繁使用“动作阶段”“打断关系”“输入权限”“状态标签”等共通语义。为了避免后续各子系统各自定义一套规则，后续建议将这些内容单独抽成一份公共规则文档；当前第 2 章先作为这些公共规则的上游需求来源。

### 2.3 受攻击与受击反馈

- 受攻击与受击反馈模块用于处理角色或敌人在被攻击命中后的伤害结算、受击状态切换、动作打断与表现反馈。当前框架下，受攻击响应同样纳入 GAS 体系管理。
- 当前所说的“受攻击技能”，指的是角色身上用于响应来袭攻击的受击技能集合。只有当角色持有与来袭攻击类型相匹配的受击技能时，角色才会在该次命中下进入对应受击状态并播放相应受击表现。
- 受击技能按攻击类型进行配置，而不是按单个攻击动作逐个手写配置。举例来说，普通攻击或部分技能攻击段可以被配置为【轻攻击】类型；如果目标角色身上具备对应的【轻攻击受击】受击技能，则在该次命中成立时，目标会进入轻攻击受击表现流程。
- 为避免后续配置语义混乱，当前框架下至少要将“受击反应类型”与“伤害/机制标签类型”拆开定义。前者用于决定命中后触发哪一类受击响应，例如【轻攻击】【重攻击】【击飞攻击】；后者用于描述该攻击的伤害属性、命中形态与特殊机制，例如【物理攻击】【法术攻击】【投射物】【可格挡】【可闪避规避】等。两者允许在同一攻击段上并存，但不视为同一层配置。
- 攻击段负责决定本次命中触发哪一种受击类型；受击技能则负责决定该类型命中后是否打断当前动作、播放什么受击表现、以及进入什么恢复流程。
- 是否进入受击状态，与是否造成伤害不是同一个判定层。也就是说，即使目标没有与本次攻击类型相匹配的受击技能，仍然可以正常进入伤害结算流程；只是该目标不会因此进入对应的受击状态或播放该类受击动画。
- 当目标成功进入受击流程后，需要先进行伤害结算，再根据对应受击技能配置决定后续表现。当前阶段要求至少支持：角色受到伤害、角色进入受击状态、角色从受击状态恢复，以及角色受到攻击时的基础反馈。
- 受击动作是否打断目标当前动作，不由攻击者单方面决定，而是由最终被触发的受击技能配置决定。不同受击技能可以分别配置为可打断或不可打断，从而允许不同敌人或角色对同类攻击表现出不同的抗打断能力。
- 闪避无敌帧规避成功的攻击不视为受击。也就是说，如果一次攻击在闪避无敌帧期间被成功规避，则该次命中不进入受击流程，不播放受击动画，也不触发受击状态。
- 格挡成功同样不视为受击。格挡成功属于独立的格挡结算流程，不纳入普通受击判定；因此格挡成功不会触发常规受击状态或常规受击动画。
- 受击方向需要根据攻击者相对于被攻击者的击中方向来判断。受击成立后，角色或敌人应根据来袭方向播放不同方向的受击动画，以保证反馈与战场空间关系一致。
- 受击状态恢复按自动结束处理。也就是说，当角色进入某种受击状态后，其恢复流程由对应受击技能自身的时长与流程控制，在受击表现播放完成或对应状态结束后，角色自动退出受击状态并回到后续可继续执行的状态。
- 当前阶段，受击模块需要与普通攻击、技能、闪避、格挡和战斗状态模块保持统一接口关系：攻击段提供攻击类型与命中信息，受击系统决定是否进入受击表现，伤害系统负责数值结算，状态系统负责记录当前受击状态与恢复结果。

### 2.4 普通攻击

- 游戏内后续可能存在多个可玩角色，每个角色都默认具备普通攻击能力，并拥有一套默认解锁的普通攻击流程。普通攻击能力默认具备，但角色在战斗中可能因受限状态而暂时无法释放普通攻击。
- 普通攻击按角色独立配置，不按武器类型复用。每个角色装备的武器类型可以固定，但即使使用相同武器的角色，普通攻击流程、表现和连段内容也可以不同。
- 每个角色的普通攻击由若干段组成，当前要求每个角色至少具备 3 段普通攻击，后续允许扩展到更多段数。
- 每一段普通攻击都视为一个独立的 GAS 技能，并通过 GAS 控制该段攻击的表现、持续时间、段间衔接关系、命中后的敌方反应，以及伤害、Buff 与 GE 结算。
- 每个角色的攻击都需要绑定对应动画，玩家点击普通攻击后会播放对应动画。每一段普通攻击可以对应一个或多个动画，也可以对应多个 Montage Section（动画蒙太奇分段）或多个动画资源节点。需要注意，一段普通攻击本身可能包含若干个动画片段。
- 普通攻击的不同段可以具备不同的位移特征，是否带位移、位移如何发生，允许按具体段单独配置。
- 每一段普通攻击都必须具备独立的属性标签，用于标识该段攻击的性质与特征。同一段普通攻击可以同时拥有多个属性标签。属性标签既用于描述攻击方式，也用于描述伤害类型；例如【投射物】既可以作为攻击方式标签，也可以作为伤害类型标签。
- 为了保证后续策划配置与程序判定稳定，普通攻击相关标签当前至少拆分为以下两层：
  1. 受击反应类型：用于决定命中后触发哪类受击，例如【轻攻击】
  2. 伤害/机制标签：用于描述伤害性质、命中来源与特殊判定，例如【物理攻击】【法术攻击】【投射物】【可格挡】【可闪避规避】
- 普通攻击的属性标签会参与后续伤害、效果和加成判定。最常见的伤害/机制标签是【物理攻击】，表示这段攻击在伤害属性上属于物理类型；如果后续某个技能、属性或效果规定对【物理攻击】生效，则应作用于带有该标签的普通攻击段。
- 不同段普通攻击允许配置不同标签组合。例如某个角色有 3 段普通攻击，第一剑可以是【物理攻击】，第二剑可以是【物理攻击】【法术攻击】，第三剑可以是【物理攻击】【投射物】。
- 普通攻击既可以因各类加成改变数值，也可以因个别效果改变技能形态。例如玩家的第三剑原本是飞出两道剑气对正前方造成伤害，后续如果存在“投射物数量增多 2”这类效果，则该段普通攻击的投射物数量会增加。
- 形态变化属于普通攻击系统需要支持的能力，但当前框架文档不约束具体实现方式，具体实现将在普通攻击详细文档阶段再与主程序确认。
- 每一段普通攻击都需要区分前摇和后摇，并允许按段分别配置。后续闪避、技能、结印等系统的打断规则，将基于前摇与后摇状态进行判定，例如闪避只能打断后摇，技能和结印可以配置为打断前摇或后摇。
- 每一段普通攻击都可能具有独立的收招逻辑。玩家如果在某个普通攻击连段后没有继续输入，则角色会根据当前段的收招流程返回待机状态，不同段返回待机状态的动作过程可以不同。
- 普通攻击命中后，不同目标是否进入受击状态，需要继续读取本章 2.3 中定义的受击类型匹配规则；也就是说，普通攻击命中成立，不等于目标一定播放受击动作。
- 当前阶段，普通攻击不考虑因长按或其他输入形式而产生派生招式。

### 2.5 预输入

- 在玩家执行上一个指令结束前，玩家输入的下一个指令会存储到预输入中。
- 上一个指令结束，当前主规则改为以“动作逻辑结束事件”为优先判定标准；动画播放完成仍然是最常见的触发来源之一，但不再作为唯一标准。也就是说，后续允许出现“逻辑上已结束、可进入下一动作，但表现层收招动画仍在播放”的情况。
- 预输入不是只服务于普通攻击连段，而是服务于所有允许进入缓存的战斗输入。例如角色正在释放一个技能时，玩家输入下一个技能指令，如果技能不能打断当前技能，则该输入会形成一个预输入。
- 只有在当前输入无法打断正在执行的指令时，输入才会进入预输入；如果当前输入可以打断正在执行的指令，则应立即执行，不进入预输入。例如角色正在进行普通攻击时，玩家输入结印指令，如果结印可以打断普通攻击，则角色应立刻执行结印输入。
- 预输入只记录最后一个合法输入，不记录多个输入后再筛选。后续新输入允许覆盖旧的预输入。
- 合法预输入至少需要同时满足两层条件：第一，这个输入本身属于允许进入预输入的输入类型；第二，在当前动作结束时，该输入仍然处于可执行状态。
- 可执行状态包括但不限于：角色当前状态允许执行该输入、资源条件满足、相关限制状态未阻止该输入、该输入在当前上下文中可正常生效。
- 不是所有输入都默认支持预输入，哪些输入允许进入预输入，需要按指令类型单独配置。
- 预输入存在一个缓存窗口。当玩家当前指令结束后，系统需要根据预输入窗口时间，检查是否存在最后一个合法的预输入指令。
- 预输入窗口时间默认配置在角色身上，不同角色可以拥有不同的预输入窗口时间；同时允许具体动作覆盖角色默认的预输入窗口时间。
- 例如角色 A 的预输入窗口是 0.4 秒，则角色当前动作结束时，需要检查前 0.4 秒内是否存在已存储的最后一个合法预输入；如果存在，则执行该输入；如果不存在，则不执行预输入指令。
- 当前动作结束后，如果执行了预输入，则本次预输入需要立刻清空。
- 当前动作结束后，如果预输入在结束瞬间已不满足可执行条件，则该预输入直接丢弃，不回退尝试更早的输入。
- 如果当前动作被更高优先级行为中断，例如被受击打断、被闪避取消、被技能取消、被其他强制中断行为终止，则应立即清空当前预输入，不在该次中断后继续执行原预输入。
- 预输入指令当前不考虑长按、按下与松开差异等输入形式。

### 2.6 闪避

- 双心印当前采用无敌帧形式的闪避。闪避成立的核心依据是角色是否处于闪避无敌帧内，而不是单纯依赖位移离开攻击判定区域，因此本项目中的闪避不采用“位移闪避判定”作为核心规则。
- 闪避本身是一个独立的 GAS 技能。角色默认具备闪避能力，但是否能够释放闪避，需要同时满足当前状态允许、未被限制标签禁止、且闪避不处于冷却中的条件。
- 闪避动作一定包含位移，但不强制所有角色共用同一种动画表现。系统层面只要求角色能够对 8 个方向输入作出对应闪避响应，具体表现可以因角色不同而不同，例如翻滚、滑步或其他形式。
- 闪避通过玩家主动按键输入触发，采用按下即触发的方式，不支持长按判定，也不区分按下与松开的不同派生逻辑。
- 闪避方向取决于玩家当前的移动输入方向；如果玩家触发闪避时没有输入移动方向，则角色默认朝当前面朝方向进行闪避。当前阶段，锁定敌人与非锁定状态下共用同一套闪避方向判定规则。
- 闪避支持预输入，并且严格遵循本章 2.5 的通用预输入规则。也就是说，闪避输入本身属于允许进入预输入的输入类型；当闪避输入无法立即打断当前动作时，可以先进入预输入缓存；是否最终执行，则在当前动作结束、预输入结算时再按通用可执行条件判断。
- 闪避与其他动作之间的打断关系通过 GAS Tag 配置，由策划配置“闪避可以打断哪些动作”以及“哪些动作可以打断闪避”。当前默认规则如下：
  1. 普通攻击的后摇可以被闪避打断
  2. 技能的后摇可以被闪避打断
  3. 各类动作的前摇默认不能被闪避打断
  4. 前摇不可被闪避打断属于默认规则，允许通过 Tag 配置为个别动作开放特例
- 闪避动作本身当前不再额外区分前摇和后摇，而是视为一个完整动作。闪避动作执行期间，玩家不能通过其他主动行为中断当前闪避；如果玩家在闪避过程中输入普通攻击、技能、结印、格挡等其他指令，这些输入会按通用规则进入预输入，而不会立即打断闪避。
- 闪避结束后，角色允许立刻衔接后续动作，包括但不限于普通攻击、技能、结印、格挡等，并且闪避结束后保留玩家当前的移动输入惯性。
- 闪避只能在非受限状态下释放。受限状态不通过固定枚举硬编码定义，而是通过“禁止释放闪避”的 GAS Tag 集合统一表达，例如眩晕、倒地等状态都可以通过持有相关限制标签来禁止闪避。
- 角色在闪避动作的若干帧内会进入闪避状态，该状态下角色视为具有无敌帧。无敌帧长度允许作为养成与角色差异的一部分进行配置，不同角色即使使用相同类型的闪避动作，也可以拥有不同的无敌帧长度。
- 闪避无敌帧可规避哪些命中与效果，后续通过 GAS Tag 交由策划配置。当前阶段的初步方向为：
  1. 可以规避：直接伤害、投射物命中、近战命中、控制效果、击退、击飞
  2. 不能规避：角色身上的持续伤害、场地上的持续伤害、标记、锁定等效果
- 当角色在闪避无敌帧期间，被一个“本应可被闪避规避”的命中实际命中时，本次行为判定为一次“闪避成功”。闪避成功时，角色不会进入本章 2.3 定义的受击流程，也不会承受该次命中对应的受击反馈。
- 单次闪避动作的“闪避成功”收益按一次计算。也就是说，一次闪避动作即使在无敌帧期间规避了多段本应可被闪避规避的命中，也只记为一次闪避成功，并只结算一次“闪避成功”相关收益，例如正式版中的资源回复。
- 如果角色在闪避无敌帧期间没有发生任何“可被闪避规避”的命中，则本次闪避不判定为闪避成功。也就是说，单纯在原地完成一次闪避动作，或在无敌帧期间未遭遇有效命中时，不会获得“闪避成功”判定。
- 如果角色在闪避动作仍未结束，但已经脱离无敌帧的阶段被敌方命中，则角色按正常规则受击：正常承受可承受的伤害与效果、进入受击状态，并由受击中断当前闪避动作。
- 闪避具有独立冷却时间，冷却规则通过 GAS 配置。闪避处于冷却期间时，角色无法再次直接释放闪避；但如果闪避输入发生在当前动作尚未结束的阶段，该输入仍可按 2.5 的预输入规则进入缓存，并在预输入结算时再次检查闪避是否已恢复可执行。
- 当前版本中，闪避不消耗任何资源，也不回复任何资源；但正式版需要为闪避预留资源接口。正式版的资源规则当前定义为：每次闪避固定消耗一种资源；如果本次闪避被判定为“闪避成功”，则回复某种资源，且回复的资源类型可以与闪避消耗的资源类型不同。

### 2.7 格挡

- 格挡由玩家主动输入按键触发，不支持长按，也不区分按下与松开的不同输入形态。格挡本身是一个独立的 GAS 技能，并且属于允许进入 2.5 预输入系统的输入类型。
- 格挡不是对“整个技能”进行统一判定，而是针对敌方某一段普通攻击，或某一段技能攻击，分别配置其是否可被格挡。是否可格挡由策划通过配置决定，并通过对应提示、判定窗口与后续结算共同生效。
- 当玩家输入格挡后，角色不会立刻完成一次硬判定，而是进入一个短暂的格挡判定窗口。只有当该窗口内，敌方存在一段可被格挡的攻击，并且角色同时满足位置条件与时机条件时，才会成立本次格挡成功。
- 格挡成功需要同时满足以下条件：
  1. 当前输入时机落在该攻击段允许被格挡的判定窗口内
  2. 当前攻击段本身被配置为可格挡
  3. 格挡者位于该次攻击允许格挡的合理位置内
- 所谓合理位置，当前至少由“攻击正面的角度范围”和“允许格挡的距离范围”共同定义。实现基准上，应优先以“攻击段自身的可格挡判定区域”作为判定依据，而不是只用攻击者朝向做粗判；攻击者朝向可以作为配置该判定区域时的常用参考，但不应替代攻击段自身的实际格挡区域。这里的距离范围需要与格挡成功后格挡动作的自动位移能力匹配，保证角色能够通过格挡动画自然移动到交汇点，而不会产生不合理的传送感。
- 格挡的表演性质高于纯时机博弈。玩家进行格挡的主要目标，不是单纯无伤抵消攻击，而是触发后续防守反击价值，例如解锁格挡后的后续技能、获得回复资源效果等。当前阶段，这些格挡成功后的收益默认为所有成功格挡都具备。
- 在攻击可被格挡的前提下，攻击者会在攻击前给出光效提示。当前规则为：
  1. 如果该次攻击段可被格挡，则提示光效为黄色
  2. 如果该次攻击段不可被格挡，则提示光效为红色
- 光效提示会同时发送给两名玩家。两名玩家只要各自满足格挡时机与位置条件，都可以主动尝试对同一攻击段进行格挡。
- 即使面对的是红光提示的不可格挡攻击，玩家仍然可以主动输入格挡。此时角色依旧会播放格挡动作，但本次格挡不会成立；如果角色在动作期间被该攻击命中，则按正常受击流程处理。
- 格挡动作本身不提供无敌帧。只有当角色在正确时机、正确位置上，与可格挡攻击段成功匹配时，才会进入格挡成功结算；否则格挡动作仅视为一次普通主动动作，不具备闪避式的无敌保护。
- 当格挡成功后，格挡者会进入格挡状态，并在短时间内失去角色控制能力。格挡成功后，角色开始播放格挡动画，格挡动画本身带有位移效果，并会控制玩家移动到格挡位置。
- 格挡位置定义为攻击者动画与格挡者格挡动画的交汇点。该交汇点需要满足一个核心表现目标：当双方动作到达该点时，视觉上应当能够清晰表现出“格挡者确实挡住了攻击者的这一次攻击”。
- 当格挡失败，即玩家输入了格挡但未在判定窗口内成功匹配到一次有效格挡时，角色依旧会播放格挡动作；但这次失败动作不消耗格挡的冷却与资源，也不会让角色进入完整的失控格挡状态。失败格挡动作期间，玩家允许通过更高优先级行为手动取消该动作，例如用闪避取消失败格挡。
- 格挡的冷却与资源消耗只在“格挡成功”成立后才会扣除，而不是在进入格挡判定窗口时预先扣除。
- 格挡成功后，需要根据攻击者与格挡者双方属性判断格挡结果。当前至少分为两层结算：
  1. 先判断格挡后的位移结果
  2. 再判断格挡后的伤害结果
- 位移结果当前按以下规则处理：
  1. 如果攻击者本次攻击的重量属性大于格挡者的格挡能力值属性，则格挡者被击退，并播放对应击退动画
  2. 如果攻击者本次攻击的重量属性小于格挡者的格挡能力值属性，则攻击者被击退，形成弹反效果
  3. 如果攻击者本次攻击的重量属性等于格挡者的格挡能力值属性，则双方都不后退
- 当双方都不后退时，本次格挡仍然视为成功，但不强制打断攻击者后续攻击连段。也就是说，如果攻击者当前设计上允许衔接后续动作，则其可以直接进入下一动作。
- 格挡成功后的伤害结算需要与格挡者的格挡能力值属性相关。当前阶段先明确方向：格挡成功并不等于完全免伤，而是根据格挡者的格挡能力值，去计算本次实际承受的伤害；具体伤害公式后续再单独确定。
- 当前单人阶段与近期双人实现阶段，若同一攻击段被两名玩家几乎同时格挡成功，系统先只认第一个成功者，不直接进入联合格挡结算。联合格挡仍然保留为后续可扩展方向，但当前框架只要求预留对应状态位或接口，不要求本阶段立即实现双成功升级逻辑。
- 格挡成功并到达双方动作交汇点后，需要预留额外演出与反馈接口，用于承载震屏、镜头倾斜等后处理表现。当前阶段只要求预留这些反馈入口，不在本框架文档中展开具体表现清单与参数。

## 3. 技能系统模块（结印系统）

### 3.1 结构定义

当前框架下，结印系统至少由以下层级组成：

1. 印
2. 招式
3. 辅助效果
4. 释放流程
5. 相关资源

### 3.2 印

玩家在战斗中先进入某个印，再从该印下选择或触发对应招式。

每个印需要满足以下结构要求：

1. 用于归类招式
2. 可承载该印上的辅助效果
3. 可作为后续合印的组合基础

### 3.3 招式

同一种印下可以存在多个招式，不同招式具备不同的战斗功能，例如：

1. 单体输出
2. 群体输出
3. 位移攻击
4. 控制
5. 辅助

### 3.4 辅助效果

辅助效果附着在印上，不直接构成独立招式，而是对该印下后续释放的招式产生统一加成或功能修正。

### 3.5 结印释放流程

当前框架下，结印系统的基本流程定义为：

1. 玩家满足释放条件
2. 玩家选择或进入某个印
3. 系统记录当前印状态
4. 玩家在该印下选择或触发某个招式
5. 招式读取当前印与辅助效果信息后生效
6. 技能消耗资源并进入对应结算流程

### 3.6 结印系统要求

结印系统需要满足以下要求：

1. 操作逻辑清晰，玩家能理解自己当前处于哪个印
2. 招式选择过程不能过度拖慢战斗节奏
3. 印、招式、辅助效果的关系必须稳定且可扩展
4. 未来需要支持双人协同影响，因此印状态不能只按单次按钮表现来设计
5. 需要适配 GAS 体系下的统一技能、状态、效果管理方式

## 4. 合印系统模块

### 4.1 合印的基础结构

合印系统在结构上至少包含以下部分：

1. 合印前提条件
2. 合印触发行为
3. 合印成功结果
4. 合印后的强化结算
5. 合印资源消耗
6. 合印失效条件

### 4.2 合印前提条件

合印至少需要依赖以下信息：

1. 两个角色都处于可参与合印的状态
2. 双方满足位置或距离要求
3. 双方拥有可参与组合的印或印状态
4. 双方满足资源要求

### 4.3 合印触发行为

后续设计中需要具备：

1. 玩家发起合印的明确操作
2. 双方角色执行合印动作或确认流程
3. 系统判定合印是否成立

### 4.4 合印成功结果

合印成功后，双方至少会发生以下变化：

1. 形成一次合印成功状态
2. 双方当前印上的部分效果进入可联动状态
3. 后续某一次技能释放获得额外强化

当前按以下规则处理：

1. 合印后，双方印上的效果会对后续技能产生叠加影响
2. 强化通常作用于下一次符合条件的技能释放
3. 强化收益可以来自招式本身、辅助效果、附加效果等多个维度

### 4.5 合印资源消耗

根据已有项目设定，合印资源消耗按以下原则处理：

1. 合印资源消耗应显著高于普通结印
2. 合印不能成为常驻最优解
3. 合印资源规则需要可配置

### 4.6 合印系统要求

合印系统需要满足以下要求：

1. 必须体现双人配合价值
2. 必须与结印系统天然联动，而不是独立外挂系统
3. 必须具备明确的成功条件、失败条件、收益结果
4. 必须支持未来联机环境中的状态同步与判定
5. 即使当前只做单人阶段，也要在架构上预留合印相关接口与状态位

## 5. 战斗数据与状态模块

### 5.1 当前需要统一管理的核心信息

当前框架阶段，至少需要统一表达以下信息：

1. 角色基础生命相关信息
2. 角色当前战斗状态
3. 普通攻击相关状态
4. 闪避相关状态
5. 格挡相关状态
6. 受击相关状态
7. 当前结印状态
8. 当前技能资源状态
9. 未来用于合印的协同状态
10. 增益减益类状态

### 5.2 状态设计要求

状态设计需要满足以下要求：

1. 状态定义统一，避免多个模块重复定义同一含义
2. 状态之间的覆盖、并存、打断关系需要可整理
3. 当前单人阶段可本地验证，未来应可拓展到联机同步
4. 与 GAS 体系中的能力、效果、标签表达保持一致方向

## 6. 单人阶段验收标准

由于当前未开发双人联机功能，因此本阶段验收只针对“单角色战斗框架是否成立”进行判断。

### 6.1 基础战斗验收

1. 角色可以稳定进入战斗状态
2. 角色可以正常执行普通攻击
3. 角色可以正常执行闪避
4. 角色可以正常执行格挡
5. 角色受到攻击时能够产生基础受击与数值反馈

### 6.2 结印系统验收

1. 角色可以进入某个印状态
2. 系统可以识别角色当前所处印状态
3. 角色可以在印下成功释放招式
4. 印上的辅助效果能够参与招式结算
5. 结印、招式、资源、状态之间的基础流程能够跑通

### 6.3 合印系统框架验收

由于当前无双人联机功能，本阶段不验收完整合印玩法效果，但至少需要验证：

1. 系统中已存在合印相关状态位或接口预留
2. 结印系统结构没有阻断未来合印接入
3. 技能与状态结构允许未来读取另一角色的协同信息
