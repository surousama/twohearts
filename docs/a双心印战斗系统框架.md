# 《双心印》战斗系统框架

本文档用于给主程序明确《双心印》战斗系统的框架内容：当前以单角色本地验证为验收口径，但整体结构必须服务未来双人共斗与联机扩展，且不跳出既定 `GAS` 技术路线。本文档只定义战斗系统的模块划分、结构关系、状态与验收边界，不展开具体角色、具体招式数值、具体特效资源与具体代码实现。

## 1. 战斗系统模块划分

当前战斗系统框架拆分为四个一级模块：

1. 基础战斗模块
2. 技能系统模块（结印系统）
3. 合印系统模块
4. 战斗数据与状态模块

其中：

- 基础战斗模块负责玩家最常使用的动作循环
- 技能系统模块负责角色主动战斗能力的核心差异化
- 合印系统模块负责双心印最核心的双人协同特色
- 战斗数据与状态模块负责支撑上述系统运行的统一状态表达

## 2. 基础战斗模块

### 2.1 模块目标

基础战斗模块用于支撑玩家最基本的动作体验，是所有技能与协同玩法的底座。

### 2.2 模块组成

基础战斗模块当前包含以下内容：

1. 普通攻击
2. 预输入
3. 闪避
4. 格挡
5. 受攻击与受击反馈

### 2.3 普通攻击

- 游戏内后续可能存在多个可玩角色，每个角色都默认具备普通攻击能力，并拥有一套默认解锁的普通攻击流程。普通攻击能力默认具备，但角色在战斗中可能因受限状态而暂时无法释放普通攻击。
- 普通攻击按角色独立配置，不按武器类型复用。每个角色装备的武器类型可以固定，但即使使用相同武器的角色，普通攻击流程、表现和连段内容也可以不同。
- 每个角色的普通攻击由若干段组成，当前要求每个角色至少具备 3 段普通攻击，后续允许扩展到更多段数。
- 每一段普通攻击都视为一个独立的 GAS 技能，并通过 GAS 控制该段攻击的表现、持续时间、段间衔接关系、命中后的敌方反应，以及伤害、Buff 与 GE 结算。
- 每个角色的攻击都需要绑定对应动画，玩家点击普通攻击后会播放对应动画。每一段普通攻击可以对应一个或多个动画，也可以对应多个 Montage Section（动画蒙太奇分段）或多个动画资源节点。需要注意，一段普通攻击本身可能包含若干个动画片段。
- 普通攻击的不同段可以具备不同的位移特征，是否带位移、位移如何发生，允许按具体段单独配置。
- 每一段普通攻击都必须具备独立的属性标签，用于标识该段攻击的性质与特征。同一段普通攻击可以同时拥有多个属性标签。属性标签既用于描述攻击方式，也用于描述伤害类型；例如【投射物】既可以作为攻击方式标签，也可以作为伤害类型标签。
- 普通攻击的属性标签会参与后续伤害、效果和加成判定。最常见的标签是【物理攻击】，表示这段攻击是物理类型；如果后续某个技能、属性或效果规定对【物理攻击】生效，则应作用于带有该标签的普通攻击段。
- 不同段普通攻击允许配置不同标签组合。例如某个角色有 3 段普通攻击，第一剑可以是【物理攻击】，第二剑可以是【物理攻击】【法术攻击】，第三剑可以是【物理攻击】【投射物】。
- 普通攻击既可以因各类加成改变数值，也可以因个别效果改变技能形态。例如玩家的第三剑原本是飞出两道剑气对正前方造成伤害，后续如果存在“投射物数量增多 2”这类效果，则该段普通攻击的投射物数量会增加。
- 形态变化属于普通攻击系统需要支持的能力，但当前框架文档不约束具体实现方式，具体实现将在普通攻击详细文档阶段再与主程序确认。
- 每一段普通攻击都需要区分前摇和后摇，并允许按段分别配置。后续闪避、技能、结印等系统的打断规则，将基于前摇与后摇状态进行判定，例如闪避只能打断后摇，技能和结印可以配置为打断前摇或后摇。
- 每一段普通攻击都可能具有独立的收招逻辑。玩家如果在某个普通攻击连段后没有继续输入，则角色会根据当前段的收招流程返回待机状态，不同段返回待机状态的动作过程可以不同。
- 当前阶段，普通攻击不考虑因长按或其他输入形式而产生派生招式。

### 2.4 预输入

- 在玩家执行上一个指令结束前，玩家输入的下一个指令会存储到预输入中。
- 上一个指令结束，通常以当前动画播放完成作为标志。
- 预输入不是只服务于普通攻击连段，而是服务于所有允许进入缓存的战斗输入。例如角色正在释放一个技能时，玩家输入下一个技能指令，如果技能不能打断当前技能，则该输入会形成一个预输入。
- 只有在当前输入无法打断正在执行的指令时，输入才会进入预输入；如果当前输入可以打断正在执行的指令，则应立即执行，不进入预输入。例如角色正在进行普通攻击时，玩家输入结印指令，如果结印可以打断普通攻击，则角色应立刻执行结印输入。
- 预输入只记录最后一个合法输入，不记录多个输入后再筛选。后续新输入允许覆盖旧的预输入。
- 合法预输入至少需要同时满足两层条件：第一，这个输入本身属于允许进入预输入的输入类型；第二，在当前动作结束时，该输入仍然处于可执行状态。
- 可执行状态包括但不限于：角色当前状态允许执行该输入、资源条件满足、相关限制状态未阻止该输入、该输入在当前上下文中可正常生效。
- 不是所有输入都默认支持预输入，哪些输入允许进入预输入，需要按指令类型单独配置。
- 预输入存在一个缓存窗口。当玩家当前指令结束后，系统需要根据预输入窗口时间，检查是否存在最后一个合法的预输入指令。
- 预输入窗口时间默认配置在角色身上，不同角色可以拥有不同的预输入窗口时间；同时允许具体动作覆盖角色默认的预输入窗口时间。
- 例如角色 A 的预输入窗口是 0.4 秒，则角色当前动作结束时，需要检查前 0.4 秒内是否存在已存储的最后一个合法预输入；如果存在，则执行该输入；如果不存在，则不执行预输入指令。
- 当前动作结束后，如果执行了预输入，则本次预输入需要立刻清空。
- 当前动作结束后，如果预输入在结束瞬间已不满足可执行条件，则该预输入直接丢弃，不回退尝试更早的输入。
- 如果当前动作被更高优先级行为中断，例如被受击打断、被闪避取消、被技能取消、被其他强制中断行为终止，则应立即清空当前预输入，不在该次中断后继续执行原预输入。
- 预输入指令当前不考虑长按、按下与松开差异等输入形式。

## 3. 技能系统模块（结印系统）

### 3.1 结构定义

当前框架下，结印系统至少由以下层级组成：

1. 印
2. 招式
3. 辅助效果
4. 释放流程
5. 相关资源

### 3.2 印

玩家在战斗中先进入某个印，再从该印下选择或触发对应招式。

每个印需要满足以下结构要求：

1. 用于归类招式
2. 可承载该印上的辅助效果
3. 可作为后续合印的组合基础

### 3.3 招式

同一种印下可以存在多个招式，不同招式具备不同的战斗功能，例如：

1. 单体输出
2. 群体输出
3. 位移攻击
4. 控制
5. 辅助

### 3.4 辅助效果

辅助效果附着在印上，不直接构成独立招式，而是对该印下后续释放的招式产生统一加成或功能修正。

### 3.5 结印释放流程

当前框架下，结印系统的基本流程定义为：

1. 玩家满足释放条件
2. 玩家选择或进入某个印
3. 系统记录当前印状态
4. 玩家在该印下选择或触发某个招式
5. 招式读取当前印与辅助效果信息后生效
6. 技能消耗资源并进入对应结算流程

### 3.6 结印系统要求

结印系统需要满足以下要求：

1. 操作逻辑清晰，玩家能理解自己当前处于哪个印
2. 招式选择过程不能过度拖慢战斗节奏
3. 印、招式、辅助效果的关系必须稳定且可扩展
4. 未来需要支持双人协同影响，因此印状态不能只按单次按钮表现来设计
5. 需要适配 GAS 体系下的统一技能、状态、效果管理方式

## 4. 合印系统模块

### 4.1 合印的基础结构

合印系统在结构上至少包含以下部分：

1. 合印前提条件
2. 合印触发行为
3. 合印成功结果
4. 合印后的强化结算
5. 合印资源消耗
6. 合印失效条件

### 4.2 合印前提条件

合印至少需要依赖以下信息：

1. 两个角色都处于可参与合印的状态
2. 双方满足位置或距离要求
3. 双方拥有可参与组合的印或印状态
4. 双方满足资源要求

### 4.3 合印触发行为

后续设计中需要具备：

1. 玩家发起合印的明确操作
2. 双方角色执行合印动作或确认流程
3. 系统判定合印是否成立

### 4.4 合印成功结果

合印成功后，双方至少会发生以下变化：

1. 形成一次合印成功状态
2. 双方当前印上的部分效果进入可联动状态
3. 后续某一次技能释放获得额外强化

当前按以下规则处理：

1. 合印后，双方印上的效果会对后续技能产生叠加影响
2. 强化通常作用于下一次符合条件的技能释放
3. 强化收益可以来自招式本身、辅助效果、附加效果等多个维度

### 4.5 合印资源消耗

根据已有项目设定，合印资源消耗按以下原则处理：

1. 合印资源消耗应显著高于普通结印
2. 合印不能成为常驻最优解
3. 合印资源规则需要可配置

### 4.6 合印系统要求

合印系统需要满足以下要求：

1. 必须体现双人配合价值
2. 必须与结印系统天然联动，而不是独立外挂系统
3. 必须具备明确的成功条件、失败条件、收益结果
4. 必须支持未来联机环境中的状态同步与判定
5. 即使当前只做单人阶段，也要在架构上预留合印相关接口与状态位

## 5. 战斗数据与状态模块

### 5.1 当前需要统一管理的核心信息

当前框架阶段，至少需要统一表达以下信息：

1. 角色基础生命相关信息
2. 角色当前战斗状态
3. 普通攻击相关状态
4. 闪避相关状态
5. 格挡相关状态
6. 受击相关状态
7. 当前结印状态
8. 当前技能资源状态
9. 未来用于合印的协同状态
10. 增益减益类状态

### 5.2 状态设计要求

状态设计需要满足以下要求：

1. 状态定义统一，避免多个模块重复定义同一含义
2. 状态之间的覆盖、并存、打断关系需要可整理
3. 当前单人阶段可本地验证，未来应可拓展到联机同步
4. 与 GAS 体系中的能力、效果、标签表达保持一致方向

## 6. 单人阶段验收标准

由于当前未开发双人联机功能，因此本阶段验收只针对“单角色战斗框架是否成立”进行判断。

### 6.1 基础战斗验收

1. 角色可以稳定进入战斗状态
2. 角色可以正常执行普通攻击
3. 角色可以正常执行闪避
4. 角色可以正常执行格挡
5. 角色受到攻击时能够产生基础受击与数值反馈

### 6.2 结印系统验收

1. 角色可以进入某个印状态
2. 系统可以识别角色当前所处印状态
3. 角色可以在印下成功释放招式
4. 印上的辅助效果能够参与招式结算
5. 结印、招式、资源、状态之间的基础流程能够跑通

### 6.3 合印系统框架验收

由于当前无双人联机功能，本阶段不验收完整合印玩法效果，但至少需要验证：

1. 系统中已存在合印相关状态位或接口预留
2. 结印系统结构没有阻断未来合印接入
3. 技能与状态结构允许未来读取另一角色的协同信息
