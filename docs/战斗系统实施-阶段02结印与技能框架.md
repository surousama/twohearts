# 战斗系统实施-阶段02结印与技能框架

## 阶段目标

建立结印系统的最小可运行框架，让“印”真正成为技能释放的前置条件和效果上下文。

该阶段完成后，至少要实现：

- 玩家可进入结印状态
- 可选择一个印作为当前印
- UI 可根据当前印展示可用技能
- 技能释放时能读取当前印并应用对应加成

## 本阶段范围

包含：

- 当前印状态管理
- 印与技能关系建模
- 技能筛选逻辑
- 技能释放上下文
- 印提供的属性修正

不包含：

- 完整的宝石系统深度组合
- 合印共享
- 大规模技能内容扩充

## 任务拆分

### 1. 设计结印数据模型

目标：
给“印”“技能”“辅助宝石”建立清晰的数据结构。

建议任务：

- 定义印的基础属性
- 定义技能的标签和前置条件
- 定义辅助宝石的插槽与修正规则
- 决定哪些放数据资源，哪些放逻辑层

完成标准：

- 一个印的数据结构可以完整表达基础加成、插槽和限制
- 一个技能可以声明自己适配哪些前置条件

### 2. 当前印状态切换

目标：
让角色能稳定持有一个“当前印”上下文。

建议任务：

- 定义当前印的存储位置
- 定义切印操作与状态切换
- 定义无印状态
- 定义切印时是否有动作、冷却或限制

完成标准：

- 角色在运行中可稳定切换当前印
- 其他系统可以读取当前印

### 3. 结印界面与技能列表刷新

目标：
在进入结印时，显示当前可释放的技能列表。

建议任务：

- 明确 UI 从哪里拿技能列表
- 根据当前印和装备情况刷新可用技能
- 区分“不可用但已装备”和“当前可用”

完成标准：

- 进入结印时可以显示技能列表
- 技能列表会随当前印变化

### 4. 技能释放上下文

目标：
释放技能时，不只是“放一个能力”，而是带着当前印和相关修正一起进入能力执行。

建议任务：

- 定义技能释放请求的数据结构
- 将当前印、技能本体、辅助修正一起传入技能能力
- 统一主动技能和部分被动修正的入口思路

完成标准：

- 同一个技能在不同印下释放时，能力内部可读到不同上下文

### 5. 印效果修正与最小验证技能

目标：
先用少量技能跑通印效果修正。

建议任务：

- 先选 1 到 2 个印
- 先选 1 到 2 个技能
- 实现伤害类型、范围修正、异常附加中的一部分
- 用最小案例验证“同一技能在不同印下效果不同”

完成标准：

- 至少一个技能能在两个不同印下表现出不同结果

## 建议产出物

- 印的数据结构
- 技能与印关系的配置结构
- 结印状态管理逻辑
- 结印 UI 原型接口
- 至少 1 组可验证技能案例

## 验收清单

- 玩家可切换当前印
- 进入结印时可展示技能列表
- 技能是否可用会随当前印变化
- 技能释放能读取当前印修正
- 同一技能在不同印下结果不同

## 后续对话建议切分

1. 先定“印/技能/宝石”数据结构
2. 再做当前印状态管理
3. 再接最小版结印 UI
4. 再做技能释放上下文
5. 最后做 1 到 2 个样例技能验证

## 备注

这一阶段的重点不是一次性把所有印和技能都做完，而是把框架做对。只要框架对，后续扩内容会比较轻松。
