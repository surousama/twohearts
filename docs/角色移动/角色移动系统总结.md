# 角色移动系统总结

## 1. Event Graph 核心变量

| 变量名 | 含义 |
|------|------|
| `GroundSpeed` | 水平移动速度 |
| `DistanceTraveled` | 上一帧移动距离，通常为 `GroundSpeed × DeltaTime` |
| `bShouldMove` | 是否应进入移动状态 |
| `MovementDirection` | 移动方向角，范围 `-180 ~ 180` |
| `Velocity` | 速度向量缓存 |
| `Acceleration` | 加速度向量缓存 |
| `DeltaTimeX` | 帧间隔时间缓存 |
| `CharacterMovement` | 运动组件引用缓存 |
| `PredictedStopDistance` | 预测停步剩余距离 |
| `bShouldPivot` | 是否应进入 Pivot |
| `PredictedPivotDistance` | 预测折返点距离 |
| `bRightFootForward` | 由 `FootPosition` 曲线判断，用于选脚 |

## 2. 状态实现要点

### 2.1 Stop

- 使用 `Sequence Evaluator`
- 进入状态时先按方向选择 Stop 动画
- `OnUpdate` 分两段处理：
  - 仍在减速时：使用 `Distance Match to Target`
  - 低速或接近停下时：放弃距离匹配，切回普通时间推进
- `Stop -> Idle` 建议按速度退出，不依赖动画是否完整播完
- `Stop -> Cycle` 转换线必须存在，否则会出现移动中继续播 Stop 的滑步问题

### 2.2 Pivot

- 触发条件核心判断为速度方向与加速度方向相反，并且速度高于阈值
- `OnUpdate` 一般分为两段：
  - 急停阶段：使用 `Distance Match to Target`
  - 反向起步阶段：使用 `Advance Time by Distance Matching`
- 当前已定位到一个关键问题：
  - `Pivot` 极短持续、随后快速进入 `Cycle` 的主路径，不是 `Pivot -> Cycle`，而是 `Pivot -> Stop -> Cycle`
  - `Pivot -> Stop` 当前使用 `not bShouldMove`，在 Pivot 急停阶段过于敏感，容易被速度/加速度短暂抖动误触发
  - 进入 `Stop` 后又会因为 `bShouldMove == true` 很快走 `Stop -> Cycle`，表现上看就像 Pivot 只闪了一下
- 当前排查结论：
  - 把 `Pivot` 内部 `bShouldPivot == false` 分支断开后，问题仍可能出现，说明主问题不在 Pivot 后半段推进
  - 把 `Pivot -> Stop` 转换线断开后，角色会卡在 Pivot 停步姿态，不再快速进入 `Cycle`
  - 因此已基本确认是 `Pivot -> Stop -> Cycle` 连跳导致的问题
- 后续处理方向：
  - 不应直接复用普通移动状态的 `bShouldMove / not bShouldMove` 作为 Pivot 退出条件
  - `Pivot -> Stop` 需要更稳的退出条件，至少应增加速度阈值或最短保护时间
  - 需要继续对照 Lyra 的 Pivot 退出逻辑；当前怀疑 Lyra 没有我们这种直接的 `Stop -> Cycle` 抢状态路径

## 3. 选脚方案

- 当前保留的方案：基于 `FootPosition` 曲线显式选脚
- `FootPosition` 的正确语义：表示“即将迈出的脚”，不是“刚着地的脚”
- 曲线在双脚交替阶段必须平滑经过 `0`
- 运行时可直接判断：
  - `FootPosition > 0`：选 `Rfoot`
  - `FootPosition < 0`：选 `Lfoot`

## 4. 踩坑经验浓缩

1. `Distance Match to Target` 不适合在角色已经基本停下时继续使用，否则容易卡姿势或跳回错误时间点
2. `Predict Ground Movement Stop Location` 如果被误当成世界坐标使用，StopDistance 会整体失真
3. `Rewind Debugger` 适合排查状态切换、选脚、动画引用是否选错，比简单打印更有效
4. 排查 Pivot 短持续问题时，不能只盯 `Pivot -> Cycle`，要同时检查是否存在 `Pivot -> Stop -> Cycle` 的中转连跳
5. `bShouldMove` 适合普通移动状态切换，但不适合直接作为 Pivot 退出条件，Pivot 阶段很容易因为急停过程中的短暂抖动被误判

## 5. 当前保留的待办

1. 摄像机旋转误触发 `Pivot`
2. `Pivot` 垂直移动检测优化
3. `Pivot` 补齐左右脚选择逻辑
4. 重做 `Pivot -> Stop` 退出条件，避免 `Pivot -> Stop -> Cycle` 连跳
