**新增变量**：

| 变量名 | 类型 | 说明 |
|--------|------|------|
| bShouldDistanceMatchStop | Bool | HasVelocity AND NOT HasAcceleration（有速度但没有加速度 = 角色正在减速滑行） |

**OnBecomeRelevant（SetUpStopAnim）**：
```
1. 根据方向选择 Stop 动画：
   Convert to Sequence Evaluator
   → MovementDirection → FSelectDirectionalAnim(Fwd/Bwd/Left/Right 4方向) → Set Sequence

2. 判断是否需要距离匹配：
   Branch (bShouldDistanceMatchStop)

   True（有速度在减速中，正常距离匹配）：
     → Distance Match to Target
       - Sequence Evaluator: Convert Return Value
       - Distance to Target: 0.0  ← 关键！不是 PredictedStopDistance，是 0！
       - Distance Curve Name: "Distance"
     → Return

   False（已经没有速度了，直接跳到停下姿态）：
     → 不设置时间（让默认值处理）
     → Return
```

> 关键细节：OnBecomeRelevant 中 Distance Match to Target 的 Distance to Target = **0.0**。
> Distance Curve 上距离=0 对应的是动画末尾（角色完全停下的帧），
> 所以这一步实际上是让动画从"停下姿态"附近开始，然后由 OnUpdate 的距离匹配往前推。

**OnUpdate（UpdateStopAnim）**：
```
Node → Convert to Sequence Evaluator (Succeeded)

→ Branch (bShouldDistanceMatchStop)
  即：HasVelocity AND NOT HasAcceleration

  True（角色仍在减速滑行，做距离匹配）：
    → PredictGroundMovementStopLocation → 计算 StopDistance
    → Branch (StopDistance > 0)
      True：
        → Distance Match to Target
          - Distance to Target: StopDistance
          - Distance Curve Name: "Distance"
        → Return
      False：
        → 正常推进时间（Advance Time by DeltaTime）
        → Return

  False（没有速度了或有加速度，正常播放）：
    → 正常推进时间（Advance Time by DeltaTime）
    → Return
```

**Lyra Stop 结构的其他细节**：
- Lyra 实际使用了**两个 Sequence Evaluator**，通过 Layered Blend per Bone 混合（上半身/下半身分离），我们暂时只用一个
- Lyra 的方向选择区分了 Jog Stop / Walk Stop / Crouch Stop 三组，我们暂时只用 Jog Stop

#### 通用方向选择函数 FSelectDirectionalAnim
- 输入：Direction (Float) + 8 个方向的动画引用
- 逻辑：`(Direction + 202.5) / 45 → Floor → % 8 → Select`
- 映射：Index 0=B, 1=BL, 2=LL, 3=FL, 4=F, 5=FR, 6=RR, 7=BR
- 标记为**线程安全**（AnimBP 绑定函数要求）
- Start / Stop / Pivot 的 OnBecomeRelevant 中复用

#### Pivot 状态实现（基于 Lyra 实际做法）

**原理**：处理角色移动中突然反方向输入的回转动画（急停 → 转身 → 反向起步）

**触发条件**：
```
(Velocity2D · Acceleration2D) < 0  AND  GroundSpeed > 50
```


**Event Graph 新增变量**：

| 变量名 | 类型 | Category | 说明 |
|--------|------|----------|------|
| bShouldPivot | Bool | Movement | 速度与加速度点积 < 0 且 GroundSpeed > 50 |
| PredictedPivotDistance | Float | Movement | PredictGroundMovementPivotLocation 返回值的 Vector Length |

**OnUpdate 连线方案（Lyra 做法）**：
```
Node → Convert to Sequence Evaluator (Succeeded) → Branch (bShouldPivot)

True（急停阶段，速度与加速度仍然相反，还在靠近回转点）：
  → Distance Match to Target
    - Distance to Target: PredictedPivotDistance
    - Distance Curve Name: "Distance"
  → Return Node

False（反向起步阶段，速度与加速度同向，已过回转点开始反向加速）：
  → Advance Time by Distance Matching
    - Distance Traveled: DistanceTraveled
    - Distance Curve Name: "Distance"
  → Return Node
```

### 左右脚选择方案调研

#### 方案 A：Sync Group 自动相位对齐（Lyra 的做法）—— 已验证不适用

**原理**：依赖 Sync Group + Sync Marker 自动对齐脚步相位，不区分 Lfoot/Rfoot 文件。

**结论**：Lyra 之所以不需要选脚，是因为它的 Start/Stop 动画包含完整步态周期，Sync Group 可以从任意相位切入。但 GASP 的动画被拆成了独立的 Lfoot/Rfoot 文件，每个文件从第 0 帧固定以某只脚开始，Sync Group 无法解决"选错动画文件"的问题。实测确认：右脚着地时进入 Stop（Lfoot 版本），会连续迈两次右脚。

#### 方案 B：FootPosition 曲线显式选脚（ALS V3/V4 的做法）—— 确定实施

**原理**：在 Cycle（Loop）动画中预制 `FootPosition` 曲线（-1=左脚在前，1=右脚在前），运行时读取曲线值，在进入 Start/Stop/Pivot 时选择对应的 Lfoot/Rfoot 动画。

**参考来源**：
- 博客园 - ALS V3 插件分析：https://www.cnblogs.com/leoin2012/p/12107707.html
- CSDN - ALS V4 Foot Lock 详解：https://blog.csdn.net/ttm2d/article/details/118525559

**实施步骤**：

1. **给所有 Run_Loop 动画添加 FootPosition 曲线**（10个方向）
   - 左脚着地帧 = -1，右脚着地帧 = 1，中间平滑过渡
   - 参考 Sync Marker 的 L/R 标记位置打关键帧

2. **Event Graph 新增变量**
   - `bRightFootForward`（Bool）：`GetCurveValue("FootPosition") > 0`

3. **修改 Start/Stop/Pivot 的 OnBecomeRelevant**
   - 在 FSelectDirectionalAnim 调用前加 `Branch(bRightFootForward)`
   - True → 调用 FSelectDirectionalAnim 传入 8 个 Rfoot 动画
   - False → 调用 FSelectDirectionalAnim 传入 8 个 Lfoot 动画（当前已有）

### 连脚同迈修复经验总结（重要）

#### 问题本质

连脚同迈的根本原因不是 Inertialization 或 Sync Group 的 Bug，而是**选脚逻辑选错了动画**。当 Cycle 中右脚在前时进入 Stop，却选了 Lfoot 版本的 Stop 动画，动画从左脚起始姿态开始播放，视觉上就是连续迈了两次同一只脚。

#### 三个修复点

**1. FootPosition 曲线的相位定义修正**

原来的错误定义：
```
L 标记帧（左脚着地）→ FootPosition = -1
R 标记帧（右脚着地）→ FootPosition = 1
```

问题：左脚着地的瞬间，左脚已经不动了，接下来要迈的是右脚。如果此刻松手进入 Stop，应该选 Rfoot 版本。但 FootPosition = -1，判断为"左脚在前"，选了 Lfoot，选反了。

**正确的理解**：FootPosition 代表的是"即将迈出的脚"，不是"刚着地的脚"。同时更关键的是，**双脚交叉的过渡时刻曲线值必须为 0**，不能让曲线在着地帧直接从 -1 跳到 1。

修正后：重新编辑所有 Run_Loop 动画的 FootPosition 曲线，确保双脚交替时曲线值平滑过渡经过 0，着地帧和曲线峰值的对应关系正确。

**2. Stop 动画的 Sync Marker 补充**

Run_Stop 动画的前几帧缺少 Sync Marker（L/R 标记），导致 Sync Group 在过渡初期无法正确对齐脚步相位。手动为 Stop 动画的前几帧补充了 L/R Marker。

**3. 选脚逻辑简化**

Event Graph 中直接读取 `GetCurveValue("FootPosition")`：
- `> 0` → 右脚在前 → 选 Rfoot 版本
- `< 0` → 左脚在前 → 选 Lfoot 版本

#### 经验教训

1. **不要陷入固有思维**：之前一直认为问题出在 Inertialization 过冲或 Sync Group 冲突上，反复调整混合参数和 Sync Group 设置，实际根因是最基础的选脚逻辑选错了。FootPosition 曲线标的不准，导致每次都选错动画，但排查时没有往曲线本身去想。

2. **善用 Rewind Debugger**：通过 Rewind Debugger 录制回放，清楚地看到每次 Stop 都在用 Lfoot 动画，才意识到选脚逻辑没生效。这个工具比 Print String 和 ShowDebug Animation 更直观有效。

3. **不要认为某个环节不可能去掉**：之前一直觉得 Distance Matching 是必须的，不敢动。但实际 Lyra 在 `ShouldDistanceMatchStop = false`（速度很低）时，直接放弃了 Distance Matching，用 Distance Match to Target(0.0) 跳到动画末尾。灵活取舍比追求完美更重要。

4. **动画曲线的语义必须和判断逻辑严格对应**：曲线值代表什么（"刚着地的脚"还是"即将迈出的脚"）、判断条件的阈值（`> 0` 还是 `< 0`）、对应选择哪个动画版本，这三者必须完全匹配。任何一环出错都会导致选脚错误。
