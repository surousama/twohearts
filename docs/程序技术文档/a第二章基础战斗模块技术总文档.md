1. 这份文档用于从主程序视角归纳《双心印》第二章“基础战斗模块”的当前技术结论与实施顺序。
2. 它是总览与分派文档，不是直接实现文档。
3. 资深程序应先阅读本文件，再进入当前指定子文档开工。

2. 对应设计文档：[a双心印战斗系统框架](../a双心印战斗系统框架.md) 第 2 章

5. 长期目标：不阻断未来双人共斗与联机扩展

# 整体技术结论

1. 总体技术栈
   沿用 `UE5 + GAS + C++`，不跳出当前项目路线。
2. 总体落地原则
   核心判定、状态流转、输入缓存、命中与伤害统一放在 C++；
   动画资源、Montage、通知点、特效与表现层留给蓝图和数据配置。
3. 当前状态判断
   最小普通攻击闭环已经完成，可作为第二章后续模块的临时验证起点；
   但当前实现仍是过渡方案，尚未进入正式战斗架构。
4. 当前主程序结论
   下一步不建议直接先做预输入；
   AbilitySystem 基础底座已经完成最小接入；
   当前应先把普通攻击完整迁入正式 Ability 承载位置；
   在此基础上再做阶段标记、基础打断和后续预输入。

   目标：

   目标：

3. 第三步：最小预输入

4. 第四步：受击与伤害
5. 第五步：闪避
6. 第六步：格挡
7. 第七步：公共战斗语义层收束

# 当前代码状态与注意点

1. 当前普攻最小闭环没有发现阻断本地验收的严重漏洞，可视为开发中的稳定过渡版本。
2. 当前实现与长期架构不一致的地方
   普攻输入桥接已经接入 GAS；
   但普通攻击段序、Montage 播放、Timer 推进、重置和主要调试状态仍集中在 `AtwoheartsCharacter`；
   长期正式承载应迁移到普通攻击 Ability。
3. 当前必须记住的后续改造点
   当前段落推进依赖 Section 时长和 Timer；
   当前阶段先把它迁入 Ability 内部承载；
   后续进入正式阶段语义后，再统一改为动画通知或动作逻辑事件驱动。
4. 当前不宜直接叠加的内容
   不建议继续在现有 Character 级最小状态机上扩展正式预输入、打断或受击联动；
   否则后续接入 GAS、阶段标记和打断规则时会产生二次迁移。
5. 当前程序侧最小共识
   普通攻击长期按 Ability 组织；
   阶段边界以后以动画通知或动作逻辑事件为主；
   打断关系按“动作类型 + 当前阶段 + 新输入类型”判定；
   通用语义层不是当前开工项，而是后置收束项。
