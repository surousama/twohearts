# 格挡反馈与白盒验收

## Goal

作为 `05-19-chapter2-basic-combat` 之下、跟在基础格挡正式实现之后的执行子 task，补齐格挡在当前阶段最小可读反馈，并完成一次面向程序与测试的白盒验收，让 Guard 不停留在“逻辑勉强成立”，而是真正达到“能观察、能判断成功或失败、能说明剩余问题在哪”的可交付状态。

## What I already know

* 父 task 已明确：格挡不能只做逻辑存在，还需要攻击提示、命中时机和格挡成功反馈这一套最小验收口径。
* `05-25-basic-guard-implementation` 的职责是让 Guard 输入、状态、窗口和结果改写正式跑通。
* 当前项目在第二章阶段强调的是：
  先跑通最小正式战斗链路；
  局部可以接受过渡实现；
  但必须把“当前解决到哪一步、剩余问题是什么”说清楚。
* 因此，本 task 的职责不是把格挡做成最终品质，而是把可读性和白盒结论做完整。

## Requirements

* 本 task 必须补齐 Guard 相关的最小可读反馈，至少覆盖：
  攻击即将到来的提示；
  Guard 进入或 Guard 窗口存在的可观察提示；
  格挡成功反馈；
  格挡失败或未命中窗口时的可区分结果。
* 可读反馈允许采用当前阶段最低成本正式手段，例如：
  动画/蒙太奇前摇；
  Notify 驱动的特效；
  材质闪光；
  HUD / Debug Overlay；
  结构化日志。
  但至少要保证程序和测试能快速分辨“为什么这次成了 / 没成”。
* 本 task 必须给出白盒验收口径，至少覆盖：
  空闲态进入 Guard；
  正确窗口内成功格挡；
  过早输入；
  过晚输入；
  不输入直接吃招；
  连续重复测试的一致性。
* 如果当前设计允许 Guard 与普通攻击 Recovery 或 Dodge 存在关系，也应在白盒验收中覆盖这些边界情况。
* 白盒验收的结论不能只写“感觉差不多”，必须能说明：
  当前逻辑是否达标；
  当前剩余问题更偏逻辑、配置、资源还是表现。
* 若在本 task 中发现 Guard 虽有反馈，但本体逻辑并不稳定，则应回退并指向前置任务修正，而不是用反馈层掩盖基础逻辑问题。

## Acceptance Criteria

* [ ] 至少存在一套最小可读反馈，使程序和测试能观察攻击提示、Guard 状态和格挡结果。
* [ ] Guard 成功与失败在当前阶段可以被明确区分，不再只能靠猜测。
* [ ] 已形成一份可执行的白盒验收口径，并实际覆盖关键输入时机与结果差异。
* [ ] 白盒结果能说明当前格挡剩余问题主要属于逻辑、资源、配置还是表现，不把问题混写。
* [ ] 本 task 没有越界扩散到完整美术打磨、最终品质镜头语言或完整测试体系建设。

## Out of Scope

* 完整终版特效、终版音效和终版镜头表现
* 长线 QA 体系、自动化测试框架
* 完整敌人种类适配与所有攻击类型回归
* 完美格挡 / 反击 / 高阶防御机制的专项验证
* 用白盒验收替代前置逻辑修复

## Technical Notes

* 父 task：`05-19-chapter2-basic-combat`
* 前置依赖：
  `05-25-basic-guard-implementation` 已让 Guard 正式跑通；
  否则本 task 不应单独开工。
* 与前两项前置的关系：
  `05-25-minimal-hostile-attack-probe` 提供攻击可读性基础；
  `05-25-player-hit-damage-minimum-loop` 与 `05-25-basic-guard-implementation` 提供结果口径。
* 当前建议程序/测试重点确认：
  哪些反馈必须在当前阶段就看得见；
  哪些反馈先允许用日志/HUD 替代正式资源；
  哪些场景最容易出现“窗口其实错了，但只靠体感看不出来”。
* 若程序开工时仍缺信息，应优先回问：
  当前更优先做调试可读性还是尽快接近正式视觉反馈；
  白盒验收是否需要专门记录到旧测试文档或新 task 附属文档；
  当前是否已有可复用的 Guard 动画/特效资源。
* 当前允许的过渡实现边界：
  允许先用调试 HUD、日志和简化特效组合完成可读反馈；
  允许白盒先聚焦单一敌对攻击样本；
  但不允许只在口头上说“已经能挡了”，却没有可复核的结果观察口径。
* 对应上游判断来源：
  `05-19-chapter2-basic-combat/info.md` 中的“2026-05-25 格挡阶段入口评估”。
