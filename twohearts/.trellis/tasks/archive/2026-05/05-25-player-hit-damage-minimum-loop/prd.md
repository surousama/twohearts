# 玩家受击与伤害最小闭环

## Goal

作为 `05-19-chapter2-basic-combat` 之下、位于最小敌对攻击探针之后的执行子 task，建立“敌方攻击命中玩家时，玩家侧能形成最小结果闭环”的正式入口，让当前战斗系统第一次具备“外来攻击作用到玩家”的基本结果表达，为后续格挡把命中结果改写成“格挡成功 / 失败 / 穿透”提供稳定承载。

## What I already know

* 当前第二章的主顺序已经推进到：普通攻击、闪避、公共动作语义层、最小预输入完成首轮正式落地。
* 父 task 对格挡阶段的判断已经明确：
  没有“外来攻击 -> 玩家受击/伤害判定 -> 格挡改写结果”的最小闭环，就不应直接把 Guard Ability 当作下一个正式实现项。
* `05-25-minimal-hostile-attack-probe` 的职责是提供最小敌方攻击来源和明确的命中时机。
* 当前代码里尚未形成正式的玩家受击结果层，也没有看到命中 / 被格挡 / 穿透 / 失败这类结果口径落地。
* 因此，本 task 的价值是先建立“玩家侧最小结果真相源”，而不是一上来做完整伤害系统。

## Requirements

* 本 task 必须在玩家侧建立一个正式、最小、可重复验证的受击结果闭环。
* 当最小敌方攻击探针的来袭命中玩家时，系统至少应能稳定区分：
  命中成立；
  命中未成立或已失效；
  为后续格挡预留“可被改写”的命中入口。
* 本轮不强制要求完整血量数值系统，但至少要有一个明确结果出口，供日志、调试 HUD、状态反馈或后续系统读取。
* 当前建议的最小结果口径至少应包含：
  攻击来源；
  命中是否成立；
  命中时间；
  是否允许后续被格挡改写；
  本次命中的基础结果类型。
* 如果本轮确实需要最小数值占位，也允许只引入极简伤害结果字段或占位值，但不能把任务目标扩大成完整 RPG 数值系统。
* 本 task 应优先建立“玩家被攻击时的结果入口”，而不是急于加入完整硬直、击退、受击动画树、受伤音效等表现层。
* 玩家侧结果层必须能被后续 `05-25-basic-guard-implementation` 拦截或改写；
  也就是说，后续格挡不能靠重新发明一套平行命中系统来绕过本 task。
* 本 task 需要为后续测试保留足够可读性：
  受击结果至少能从日志、调试字段、HUD 或其他当前可用方式被观察到。
* 若程序在实现过程中发现“命中结果结构”和“玩家受击表现”已经开始耦合失控，应优先保证结果结构稳定，再把表现层留给后续 task。

## Acceptance Criteria

* [ ] 最小敌方攻击探针命中玩家时，玩家侧会产生一次明确、可读的受击结果。
* [ ] 该结果不是散落在临时日志里的不可复用信息，而是后续格挡可正式接入的稳定入口。
* [ ] 至少能区分“命中成立”与“未命中/已失效/未进入有效窗口”等基础差异。
* [ ] 结果中已预留后续格挡改写所需的最小语义，不需要在 Guard task 里推翻重做玩家受击入口。
* [ ] 本 task 没有越界扩散成完整血量系统、完整受击动画系统或完整敌我战斗框架。

## Out of Scope

* 完整血条、角色属性、成长数值系统
* 完整受击硬直、击飞、倒地、打断关系大重构
* 正式格挡输入、格挡窗口和格挡成功反馈
* 敌人侧的完整攻击框架重做
* 所有未来攻击类型一次性接入

## Technical Notes

* 父 task：`05-19-chapter2-basic-combat`
* 前置依赖：
  应在 `05-25-minimal-hostile-attack-probe` 已提供稳定攻击来源后开工；
  若前置任务仍未给出明确来袭事件或命中窗口，本 task 不应凭空自造另一套攻击入口。
* 后置依赖：
  `05-25-basic-guard-implementation` 应直接依赖本 task 提供的玩家受击结果入口；
  `05-25-guard-feedback-and-whitebox-validation` 也会基于本 task 的结果口径做验证。
* 当前建议程序重点确认：
  玩家侧是否需要单独的“命中结果结构”或“受击结果枚举”；
  结果是写入角色、组件还是战斗模块内的新承载点；
  当前调试 HUD 是否需要最小扩展以展示该结果。
* 当前建议优先复用或检查的代码落点：
  `Source/twohearts/twoheartsCharacter.cpp`
  `Source/twohearts/TwoHearts/Combat/TwoHeartsCombatActionContextComponent.h`
  `Source/twohearts/TwoHearts/Combat/TwoHeartsCombatActionContextComponent.cpp`
* 若程序开工时仍缺信息，应优先回问：
  这一轮是否需要最小血量占位，还是先只做结果语义；
  玩家受击结果是否希望同步到 HUD；
  当前是否已有可复用的受击动画或本轮明确先不做。
* 当前允许的过渡实现边界：
  允许先以结果结构、日志和调试展示为主；
  允许先没有正式血量 UI；
  但不允许只留下“打印了一句 hit”这种无法被后续格挡复用的伪闭环。
* 对应上游判断来源：
  `05-19-chapter2-basic-combat/info.md` 中的“2026-05-25 格挡阶段入口评估”。
