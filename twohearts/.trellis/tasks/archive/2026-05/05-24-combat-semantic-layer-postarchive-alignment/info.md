# 公共战斗语义层收尾后的现状对齐 - 主程序结论

## 当前代码现状

1. `05-21-combat-semantic-layer-readiness` 对应的首轮公共战斗语义层已经完成并归档。
2. 当前已经正式落地的能力包括：
   普攻 `1-2-3` 的 `Gameplay Ability` 单路径承载；
   基础闪避的方向解析、Montage 驱动、冷却、无敌帧、打断接入；
   公共动作上下文；
   统一输入评估结果口径 `ExecuteNow / BufferInput / Reject`；
   HUD / 日志级白盒观察口径。

## 1. 现在是否有完善的预输入系统

结论：**没有。**

当前只有“最小预输入接入口”和“统一输入评估口径”，还不是完整预输入系统。

### 已有内容

1. `UTwoHeartsCombatActionContextComponent::EvaluateInputForAction` 已经能根据当前动作上下文输出：
   `ExecuteNow`
   `BufferInput`
   `Reject`
2. 普攻在 `Startup / Active` 阶段下，重复攻击输入可以通过 `ForwardToActiveAbility` 交给当前 active ability 处理。
3. 对于更晚的缓冲时机，当前实现只会返回 `ReserveForFutureBufferConsumer`，并明确写着“这个 build 还没有消费 late buffered attack input”。

### 缺失内容

1. 没有正式的缓冲队列承载结构
2. 没有统一的“何时消费缓冲输入”机制
3. 没有覆盖 Dodge / Guard / 受击 / 技能等更完整动作集合
4. 还没有把“评估结果”和“后续实际消费”闭环成完整系统

所以当前状态应表述为：

**预输入前置条件已具备，最小接入口已留好，但完整预输入系统尚未完成。**

## 2. 现在是否具备开发格挡的条件

结论：**还不建议立刻进入格挡正式开发。**

### 已具备的前置

1. 现在已经有公共动作上下文，后续 Guard 可以接入同一套动作类型 / 阶段 / 结束原因
2. 已经有统一输入评估口径，后续 Guard 输入不必重新走 Character 私有状态机
3. Dodge 与普攻之间已经形成一条真实动作链路，说明公共语义层不是空壳

### 仍缺的关键前置

1. 预输入还未正式完成  
   格挡通常强依赖更稳定的输入缓冲 / 抢输入 / 时机判定
2. 受击与伤害系统还没形成正式承载  
   没有伤害与受击，就无法稳定定义“格挡成功 / 破防 / 挡住后的反馈”
3. 敌人系统缺失  
   没有真实来招方，格挡窗口、方向、响应都只能停在空转验证
4. 角色整体战斗手感还未完成基础收口  
   现在就做格挡，容易把临时手感问题、受击缺口和输入缺口混到一起

因此更合理的顺序仍是：

1. 最小预输入
2. 受击与伤害
3. 再进入格挡

## 3. 武器、伤害系统、敌人系统、角色手感这些内容建议什么时候补

主程序建议不要“一起补”，而是分阶段补。

### 当前阶段就该继续补的

1. **角色手感基础收口**
   包括移动、普攻、Dodge 的节奏感、响应感、切段衔接与运行态调试
2. **最小预输入**
   这是下一阶段动作输入层继续扩展的必要前置

### 预输入之后应立即补的

1. **受击与伤害系统**
   这是后续格挡、敌人交互、战斗闭环的核心基础
2. **最小敌人系统**
   不需要一开始做完整 AI，但至少要有可稳定出招、可受击、可验证玩家链路的敌对目标

### 与受击/伤害并行或稍后补的

1. **武器正式承载**
   若当前普通攻击仍主要是空手验证链路，武器建议放在“玩家战斗闭环可跑通”之后补
2. **更完整手感调优**
   在有敌人、有受击反馈后再做，调优依据会更真实

### 不建议现在就重投的大项

1. 完整格挡
2. 大而全敌人 AI
3. 完整武器系统
4. 过早追求最终动画表现层 polish

## 推荐的下一步主线

1. 收口当前普攻 + Dodge 手感
2. 完成最小预输入
3. 落最小受击与伤害
4. 补最小敌人系统，形成真实攻防验证对象
5. 再进入格挡
