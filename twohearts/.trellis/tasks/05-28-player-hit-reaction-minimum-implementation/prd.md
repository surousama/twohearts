# 玩家受击状态最小实现

## Goal

作为 `05-19-chapter2-basic-combat` 之下、建立在正式伤害结果之上的执行子 task，建立玩家受击状态、自动恢复与最小方向数据承载，让当前第二章第一次具备“受到伤害 -> 进入受击 -> 退出受击”的正式状态闭环，而不是只有伤害数字或命中结果。

## What I already know

* 原始框架文档要求至少支持：角色受到伤害、角色进入受击状态、角色从受击状态恢复，以及角色受到攻击时的基础反馈。
* 当前阶段已经具备最小敌对攻击来源、最小命中结果和即将补齐的正式伤害结果。
* 但当前还没有正式的玩家受击状态，也没有明确的恢复与打断承载。
* 如果没有这一步，后续 Guard 成功 / 失败的对照关系仍然是不完整的，因为“正常吃招”还没有成为一个正式动作状态。
* 因此，本 task 的重点是先把最小受击状态做实，而不是一次性扩展到击飞、倒地、硬直大全或完整动画树。

## Requirements

* 建立一个正式、最小、可观察的玩家受击状态入口。
* 当玩家受到一次正式伤害结果时，系统至少需要完成：
  进入受击状态；
  记录受击开始时间；
  自动恢复退出；
  留下明确的结束原因或恢复结果。
* 当前需要明确“受击是否打断当前动作”的最小规则，不允许继续依赖隐式行为。
* 当前应至少提供最小受击方向数据：
  允许本轮只做方向分类或方向向量记录；
  不要求一次性补齐完整四向 / 八向受击动画库。
* 需要保留最小可观察性：
  HUD、日志或调试字段能够说明当前是否受击、受击来源、受击方向与恢复状态。
* 本轮允许先采用极简受击表现或占位动画，但受击状态本身必须是正式规则状态，而不是“打印一句 hit react”。
* 本轮不应把任务扩大成完整控制效果系统、击退 / 击飞 / 倒地大系统或完整受击动画重定向工程。

## Acceptance Criteria

* [ ] 玩家受到一次正式伤害结果后，会进入一个正式的最小受击状态。
* [ ] 受击状态会按当前规则自动恢复退出，而不是长期残留或只靠手动清理。
* [ ] 受击期间与当前动作的最小打断关系已经明确且可观察。
* [ ] 当前已经至少存在稳定的受击方向数据承载。
* [ ] 本轮没有越界扩散成完整击退 / 击飞 / 倒地 / 控制系统或完整受击动画库建设。

## Out of Scope

* 完整受击动画资产体系
* 击退、击飞、倒地、霸体、韧性等大系统
* 完整敌人受击系统
* 完整控制效果与恢复免疫系统
* 格挡成功后的高级反击演出
