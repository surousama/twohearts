# 白盒测试计划与 PIE 跑测清单

## 测试目标

验证“最小预输入正式实施”是否满足以下最小承诺：

1. 晚到输入不再只停在预留口子，而是进入正式缓冲
2. 缓冲输入在正确时机被 Ability 或动作链路消费
3. 消费后的行为、失败回退与调试信号可观察
4. 普攻与 Dodge 至少在接口层共用同一套最小预输入口径

## 白盒检查点

1. 输入评估阶段
   检查 `ExecuteNow / BufferInput / Reject` 的分流是否仍建立在统一输入评估上，而不是回退到 Character 私有判断。

2. 缓冲承载阶段
   检查缓冲输入是否写入 `CombatActionContext`，以及记录了动作类型、消费路由与原因。

3. 普攻消费阶段
   检查普攻结束、连段衔接或晚到输入处理时，是否真的调用正式消费逻辑，而不是只写日志。

4. Dodge 衔接阶段
   检查 Dodge 完成后的后续输入是否走同一套缓冲输入承载/消费口径，或是否存在只覆盖普攻的实现缺口。

5. 失败回退阶段
   检查消费到“不属于当前链路的 buffered input”时，是否能安全放回缓冲或保持可追踪状态。

6. 调试观察口径
   检查 HUD、debug event、控制台命令是否足以区分：
   立即执行；
   已缓冲；
   已消费；
   被拒绝。

## PIE 跑测前准备

1. 启动 PIE 前确认使用带 HUD 的调试角色与控制器。
2. 打开普攻调试面板与日志。
3. 如有需要，打开 verbose 日志，便于观察晚到输入与消费时机。
4. 清空上一轮调试事件，避免旧事件干扰判断。

## PIE 跑测清单

1. 普攻立即执行
   操作：静止状态按一次普攻。
   观察：输入应直接执行；HUD 中当前 action 应进入 `NormalAttack`；不应出现残留 buffered input。

2. 普攻连段缓冲
   操作：在普攻第 1 段未到可立即衔接时，提前再按一次普攻。
   观察：第一次晚到输入应进入 buffered 状态；HUD / debug event 应能看到 buffered action type 与 reason。

3. 普攻缓冲自动消费
   操作：延续上一步，不再追加输入，等待第 1 段结束或进入允许消费时机。
   观察：缓冲输入应自动被消费并衔接到下一段；事件中应能区分“已缓冲”与“已消费”。

4. 普攻错误输入回退
   操作：在普攻过程中制造一个会被缓冲但不属于当前普攻后续消费目标的输入，如果当前版本仅支持 Dodge/普攻中的一类，也要记录实际行为。
   观察：若被当前消费者取出但不匹配，应确认是否安全回退或保留可追踪状态，而不是静默丢失。

5. Dodge 后续输入衔接
   操作：Dodge 过程中或将结束时输入普攻。
   观察：若设计目标已覆盖，应看到进入统一缓冲并在 Dodge 完成后消费；若未覆盖，应明确记录当前实际行为与证据。

6. 被拒绝输入
   操作：在明显不允许输入的时机重复触发普攻或 Dodge。
   观察：日志/HUD 应能区分这是 `Reject`，而不是误记为 buffered 或 consumed。

7. 调试信号完整性
   操作：重复以上关键路径各一次。
   观察：HUD 字段、事件名、失败原因、buffer reason 至少能支持定位“为什么执行/缓冲/消费/拒绝”。
