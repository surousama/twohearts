# 白盒测试问题单规范

## 推荐结构

1. 问题标题
2. 优先级
3. 触发条件
4. 复现步骤
5. 现象
6. 预期
7. 实际
8. 影响
9. 代码落点
10. 问题说明
11. 建议修法
12. 复测重点

## 状态标记建议

1. `已复现`：本轮确认仍存在
2. `本轮暂未复现`：历史问题本轮没再跑出来，但不能直接判已解决
3. `已解决`：已有明确修复结果，并完成最小回归验证
4. `需继续观察`：问题风险存在，但当前证据还不足以下最终结论

## 记录原则

1. 把体验问题翻译成程序可修的问题
2. 代码落点尽量具体到文件与函数附近
3. 新问题与历史问题要区分“已复现”“本轮暂未复现”“已解决”
4. 测试结论优先收敛当前主问题，不把一般性观察混成主结论
5. 现象、根因推测、修法建议和复测重点要分开写，避免混成一段
6. 若问题涉及运行态调试工具，应记录观察入口，例如 HUD 字段、日志事件名、控制台开关

## 调试观察口径

1. 优先记录当前可见的 HUD 字段、日志事件和控制台命令
2. 如果一个问题高度依赖调试面板或结构化日志才能定位，应把这些观察入口一起写进问题单
3. 测试结论里要区分“功能真的错了”和“当前只是缺少足够的观察信号”
