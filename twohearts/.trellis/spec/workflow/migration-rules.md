# Trellis 迁移期规则

## 当前迁移策略

1. `.trellis/spec` 逐步成为主真相源
2. 迁移过程中保留 `../docs/` 原文档，不删除旧内容
3. 迁移成功的判定标准是：至少一个真实任务用新 workflow 完整跑通
4. 在验证跑通前，旧 `docs` 继续作为回退资料保留
5. 验证跑通后，稳定规则优先维护在 `.trellis/spec`，`docs/` 仍可继续维护，但主要保留策划类和长期背景类文档
6. `docs/` 下的开发任务文档、程序实施过程文档、测试文档不再作为持续维护主区，以留档回看为主

## AI 行为要求

1. 迁移期间新增稳定规则，优先回写 `.trellis/spec`
2. 不要在未验证跑通前删除或覆盖旧 `docs`
3. 若发现新旧内容冲突，先指出差异，再由用户决定是否覆盖
4. 若当前任务依赖迁移前文档，可读旧 `docs`，但应优先把有效结论提炼回 `.trellis/spec`
5. 需要持续维护的策划内容可以继续写在 `docs/`，但 task、测试与开发过程文档应优先落到 `.trellis/`
